# テストマトリクス駆動開発フロー設計

## 概要

本ドキュメントは、以下の開発フローを実現するための仕組み設計をまとめたものである。

```
[ Excel（マトリクス編集・レビュー） ]
            ↓（エクスポート）
[ JSON（中間データ・Git管理） ]
            ↓（生成）
[ Gherkin（Scenario Outline） ]
            ↓
[ テストコード / 実行 ]
            ↓
[ レポート（Allureなど） ]
```

本構成の目的は以下の通り：

* 網羅性の担保（マトリクス）
* 差分管理（Git）
* 自動実行（Gherkin）
* 可視化（レポート）

---

## 全体方針

### 基本原則

* Excelは「ビュー専用」とする（Git管理しない）
* JSONを唯一の真実（Single Source of Truth）とする
* Gherkinは生成物とする（手書き禁止）
* テストコードもGherkinに従属させる

### 想定

* テストパターンはExcelで人間が作る場合もあるが、AIがJSONを直接生成するケースも想定する。
* Excelはあくまで人間のためのUIであり、JSONが機械処理のためのデータフォーマットである。
* AIが作成したテストパターンのJSONからExcelへの逆変換は行えるようにする。

---

## 各レイヤの責務

### 1. Excel（マトリクス編集・レビュー）

#### 目的

* テストパターン設計
* 網羅性確認
* レビュー

#### ルール

* 1行 = 1テストケース
* 必須列：ID
* 列名はそのまま変数名として扱う

#### サンプル

| ID    | user_type | payment | product | expected |
| ----- | --------- | ------- | ------- | -------- |
| TC001 | normal    | credit  | normal  | success  |

#### 注意事項

* Gitには保存しない
* 必ずJSONエクスポートする

---

### 2. JSON（中間データ）

#### 目的

* 差分管理
* 機械処理
* Gherkin生成の入力

#### サンプル（JSON）

```json
[
  {
    "id": "TC001",
    "user_type": "normal",
    "payment": "credit",
    "product": "normal",
    "expected": "success"
  }
]
```

#### ルール

* Gitで管理する
* PRレビュー対象とする
* スキーマを固定する

---

### 3. Gherkin生成

#### 目的

* 実行可能なテストケース生成

#### 方針

* Scenario Outlineを使用
* データはExamplesに集約

#### テンプレート

```gherkin
Feature: 商品購入

Scenario Outline: 商品購入 <id>
  Given ユーザー種別が "<user_type>"
  And 支払い方法が "<payment>"
  When 商品を購入する
  Then 結果が "<expected>" である

Examples:
  | id    | user_type | payment | product | expected |
  | TC001 | normal    | credit  | normal  | success  |
```

#### 生成ルール

* JSONのキー → Examples列
* JSONの値 → Examplesデータ
* Scenarioはテンプレ固定

---

### 4. テストコード / 実行

#### 目的

* Gherkinに基づく自動テスト

#### 構成

* Step Definitionsを実装
* Given / When / Thenをコードにマッピング

#### 例（概念）

```csharp
[Given(@"ユーザー種別が ""(.*)""")]
public void GivenUserType(string userType)
{
    // セットアップ処理
}
```

---

### 5. レポート（Allureなど）

#### 目的

* テスト結果の可視化
* 非エンジニア向け共有

#### 内容

* シナリオごとの結果
* ステップ実行ログ
* 成功 / 失敗

---

## 開発フロー

### ① マトリクス作成

* Excelで設計
* 網羅性レビュー

### ② JSONエクスポート

* Excel → JSON保存

### ③ Git登録

* JSONをコミット
* PRレビュー

### ④ Gherkin生成

* スクリプトで自動生成

### ⑤ テスト実行

* Cucumber等で実行

### ⑥ レポート生成

* Allure等で可視化

## 重要な設計ルール

### 1. 一意IDによるトレース

```
Excel行 = JSON行 = Gherkinケース
```

---

### 2. データとロジックの分離

* テンプレ（Gherkin）とデータ（JSON）を分離
* 再利用性を確保

---

### 3. Excel依存の排除

* ExcelはあくまでUI
* 真実はJSONにある
